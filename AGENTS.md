# AGENTS.md — operating guide for scheduled / future runs

This file is the **durable handoff**. The pipeline is designed so that **all state lives
in git + GitHub Issues + the vault token** — there is NO hidden state in any agent
sandbox. To resume (e.g. a daily scheduled run), an agent needs only: (a) these two
repos, (b) the GitHub PAT, (c) this file.

## Mission
Track **benchmark citations** across the LLM ecosystem. Topline entity = the **benchmark**
(not the model). For each benchmark: how often cited, source class (weight_class), citing
lab/model, methodology deviations, and whether it's already in Harbor.

## Where state lives (durable, no sandbox dependency)
All data is FLAT FILES (no data-in-code). Scripts are pure transforms.
- `christicode/benchmark-citations` — pipeline + flat-file data (this repo).
  - `labs.yaml` — in-scope labs as CONFIG (+ `source_index_urls` to crawl). Onboard a lab = edit this only.
  - `data/registry.yaml` — **single source of truth for identity**: raw→canonical aliases
    (deterministic, never fuzzy), `type` (agentic|chat), `domain`, curated collision notes.
  - `data/documents.yaml` — one entry per source document (+ `default_weight_class`).
  - `data/extractions/<lab>/<id>.yaml` — benchmark mentions per document (raw name, optional
    `weight_class` override, score, model_config, methodology). This is where extraction output goes.
  - `data/citations.jsonl` — the citation graph (GENERATED by `scripts/build.py`; never hand-edit).
  - `data/harbor_adapters.json` — generated live-Harbor snapshot (fallback cache).
  - `schema/extraction_schema.json` — the generated citation-record shape.
  - `scripts/build.py` — extractions × registry → `citations.jsonl` (pure transform).
  - `scripts/discover.py` — **the searcher**: crawl `labs.yaml` feeds + HF org API → `data/candidates.jsonl` (human-promotion queue). No extract, no issues.
  - `scripts/sync_harbor.py` — fills `harbor_*` from live registry+adapters; owns `changelog.jsonl`.
  - `scripts/scoring.py` — the ONE source-weight definition (blog_headliner 3 / model_card 2 / system_card 1).
  - `scripts/extract.py` — detection helper: doc → detected benchmarks → normalized (flags unmatched).
  - `scripts/rank.py` — per-benchmark citation SUMMARY (data-QA lens; NO priority score — deferred).
  - `scripts/build_dashboard.py` — static GitHub Pages dashboard (`docs/index.html`).
  - `scripts/review_queue.py` — live view of open `needs-human-review` issues.
- `christicode/benchmark-citations-sources` — raw source PDFs (Git LFS) + `manifest.yaml`
  (source_url + sha256 per doc). Rehydrate with `scripts/fetch_sources.py`.
- **GitHub Issues** (label `needs-human-review`) — the review queue's single source of truth.
- **Harbor**: `harbor-framework/harbor` `registry.json`, fetched **live from main** each run.
  Dedupe by `(name, version)` — names are NOT unique.

## Source scoring (weight_class)
Each mention carries a `weight_class`, decoupled from the physical document — a single blog can
have BOTH: `blog_headliner` (3, a chart/paragraph about ONE benchmark) and `model_card` (2, the
comparison table). A `system_card` row = 1 (no table-size discount). At extraction time, set a
mention's `weight_class` only when it differs from the document's `default_weight_class`. Weight
is counted as MAX per (document, model), summed across documents (3 headline blogs = 9).

## Secrets / permissions (in the vault)
GitHub fine-grained PAT on `christicode` with **Administration + Contents + Workflows +
Issues: write**, All repositories. Network access for crawling + `curl` PDF fetch
(`curl` succeeds on CDNs where the hosted web-fetch is blocked). `pdftotext` (poppler) for text.

## Daily scheduled run — procedure (feeds → candidates → human promotes → extract)
1. **Discover.** The `watch` workflow runs `scripts/discover.py` on a schedule: it crawls each
   lab's `source_index_urls` + the Hugging Face org API, keeps only URLs matching that lab's
   `candidate_patterns`, diffs vs tracked docs, and refreshes `data/candidates.jsonl`. **Enumerate
   from the index — never from a guessed list of model names** (that's how Mythos/Fable was first
   missed; discover by crawling, then diff).
2. **Diff.** `discover.py` already drops docs whose URL is in `data/documents.yaml` /
   `data/citations.jsonl`. What remains in `candidates.jsonl` = new candidates.
3. **Present candidates for HUMAN promotion.** Do NOT auto-extract sight-unseen. Open (or
   update) a candidates issue listing new docs. Wait for human promotion.
4. **On promoted docs:** archive the raw PDF to `benchmark-citations-sources` (append to
   `manifest.yaml` with sha256); run `scripts/extract.py` to detect benchmarks; read exact
   scores from tables (curated, reviewed); write a `data/extractions/<lab>/<id>.yaml` file
   (and a `data/documents.yaml` entry). Register any new benchmark name in `data/registry.yaml`.
   Any name still unmatched at build time → `needs_review` → open an issue; never guess.
   **Set `weight_class` per mention** (blog_headliner 3 / model_card 2 / system_card 1) — a blog
   can carry both a headliner chart AND a model-card table; only override when it differs from the
   doc's `default_weight_class`.
5. **Summarize.** Run `scripts/rank.py` for a per-benchmark citation data-QA summary (no priority
   score — that's deferred).
6. **Dashboard.** Regenerate `docs/` via `scripts/build_dashboard.py`. Commit.

`data/citations.jsonl` and `docs/index.html` are regenerated **server-side** by the `regenerate`
Action on push to the scripts OR the flat-file inputs (`data/registry.yaml`, `data/documents.yaml`,
`data/extractions/**`) — don't hand-push the generated `citations.jsonl` through the MCP file API.

## Guardrails (hard rules)
- **Never** `git push`, create a repo, open/close issues, or publish the dashboard without
  **explicit human confirmation**, stating exactly what will be published. Local commits are free.
- **Never fabricate** a citation or methodology detail. If a doc is ambiguous or a card is
  image-only, **flag for review** (GitHub issue) — don't guess.

## Writing GitHub issues (review queue) — house style
Terse, human, actionable — NOT AI-prose. Use these sections in order; omit any that don't
apply (use judgment). Prefer bullets over paragraphs.
1. **Problem** — what needs solving (1–3 lines).
2. **Impact** — what breaks / stays wrong if we don't do it.
3. **Options & Tradeoffs** — the real choices, each with a pro/con.
4. **Recommendation** — what Claude thinks we should do.
5. **AI instruction** — concrete steps for an agent to fix it (files/commands), where an
   agent actually *can* fix it.
- Judgment: for pure-human tasks (e.g. reading chart-image scores — see #1, #9) keep only
  **Problem / Impact / Recommendation**; drop Options and AI-instruction.
- Always stamp the opened issue's number onto the flagged citation's `review_issue` field.

## Known gotchas
- Some cards present benchmarks as **images only** (Mistral Large 3, Gemini 3 Pro/3 Flash
  model cards) → numbers must come from OCR or the blog; flag, don't invent.
- Some blogs/cards are **JS-rendered or gated** (Kimi K2.x pages, gated HF repos) → `curl`
  returns no scores; flag for headless render / human extraction, don't guess.
- **Config-dependent scores**: Fable 5 reroutes cyber/bio to Opus 4.8 (0.0%); headline
  numbers are the unblocked Mythos 5 — always record `model_config`.
- **Harness matters**: Terminal-Bench via Terminus 1 vs Terminus 2 vs Claude Code vs Codex
  gives different scores — capture it; cross-lab table columns often use different harnesses.
- **Multi-column tables**: confirm which column is the citing model before recording a score.
- **HLE / SWE-bench** etc. get restated after grader/scaffold changes — a sha256 mismatch in
  `manifest.yaml` means the upstream doc changed; re-review.
- **Large/binary files can't go through the MCP file API** — use the `regenerate` and
  `archive-sources` Actions (server-side) or a local `git`/`git-lfs` push.
